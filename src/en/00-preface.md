# Preface

## Why writing this book?

This book comes from a personal goal: to learn better, organize my thoughts, and understand software engineering at a deeper level.

But more than anything, it comes from a question I consider essential, and one we often forget:

> Why?

Why does this technology exist?
Why use this language instead of another one?
Why apply this design pattern?
Why choose microservices, a monolith, serverless, or an event-driven architecture?
Why is this solution right for this problem, and not just another way to follow the trend of the week?

In software, we talk a lot about the **what** and the **how**.

What framework to use. How to implement an API. How to deploy to the cloud. How to integrate artificial intelligence. How to scale an application.

All of that matters. But without the **why**, even a clean implementation can still be a bad decision.

To me, that is one of the biggest differences between someone who builds software and someone who thinks like an engineer. The first one can make something work. The second one asks whether it should have been built that way in the first place.

This book is built around that idea.

It is not about collecting languages, frameworks, patterns, or cloud services like trophies. It is about understanding what problem each thing solves, where it came from, what advantages it brings, what costs it introduces, and when it is actually worth using.

Because not all complexity is architecture. Sometimes it is just complexity with better branding.

With artificial intelligence, this conversation matters even more. AI can speed up how we learn, code, document, review, and build products. It is a powerful tool. Probably one of the most important tools we will see in our careers.

But it can also create a false sense of ease.

An application can work in a demo and still be insecure, expensive, slow, hard to maintain, or fragile in production. Real software does not end when it compiles, when the happy path passes, or when the UI looks good. Real software lives with users, data, costs, errors, maintenance, security, infrastructure, and consequences.

That is why fundamentals matter. A lot.

Just like in other engineering disciplines, building something that looks finished is not enough. In software, assembling pieces is not enough either. We need to understand the foundations, the risks, and the trade-offs. A poorly designed healthcare system can expose sensitive data. An insecure financial system can put real money at risk. A faulty logistics system can create losses, delays, and operational chaos.

Not every project needs the same architecture or the same level of rigor. That is also part of good judgment.

Sometimes we need strong security, auditing, traceability, and robust access control. Sometimes we need to scale to millions of users. But other times, we simply need a clear, maintainable, honest solution to the real problem.

An admin dashboard for one hundred users does not need to be born as a microservices ecosystem running on Kubernetes with three queues, five databases, and a diagram that looks like the subway map of a large city. Maybe it needs a good monolith: well-organized, tested, observable, and easy to maintain.

The main idea of this book is this:

> A true senior engineer is not the one who knows the most languages, frameworks, or technologies, but the one who understands why they exist, what problem they solve, and when it makes sense to use them.

That kind of judgment does not appear out of nowhere. It is built by studying fundamentals, making decisions, getting things wrong, operating real systems, dealing with consequences, and learning to ask better questions.

That is why this book will not be an encyclopedia or a quick guide to the framework of the month. It will be a practical path to study software engineering from its foundations and connect it to real engineering decisions.

We will talk about computer science fundamentals, memory, pointers, references, operating systems, networks, programming languages, paradigms, data structures, algorithms, software design, patterns, architecture, databases, DevOps, QA, security, infrastructure, observability, artificial intelligence, and technical leadership.

But not as an academic checklist.

Each topic should help us answer more useful questions:

* What is it for?
* Why does it exist?
* What problem does it solve?
* What cost does it introduce?
* When is it worth using?
* When is it unnecessary?
* How does it apply to real systems?

This book is mainly written for me, as a way to study, organize, and go deeper. But it is also written for other developers and engineers who want to strengthen their foundations and look at the profession with better judgment.

Because the most valuable skill in this stage of the industry will not be writing more code faster. More and more tools will be able to do that.

The valuable skill will be knowing how to guide, review, question, decide, and correct with judgment.

AI can help us build. But someone still has to know where to build, why to build it that way, and what consequences that decision may have. That someone needs foundations. Context. Responsibility.

Being senior does not mean knowing everything. It means thinking better, asking better questions, communicating better, and making decisions with more care.

This book does not pretend to have the final word. It is meant to be a map. A way to connect concepts, technologies, and decisions so we can understand the craft better.

The purpose is simple:

**To recover the art of asking why. So we can build better software, make better decisions, and grow into more complete engineers.**

---

## How to read this book

This book can be read from beginning to end, or used as a study guide.

It is not meant to be rushed. It is meant to make you pause, take notes, question decisions, look for examples, and compare each concept against real systems.

If this book makes you ask “why?” before choosing a technology, then it has already done a big part of its job.
